# Data Analysis Agent Design Spec

## Overview

A conversational data analysis agent that runs in Claude Code / Open Code. Users ask questions in natural language; the agent clarifies requirements through multi-turn dialogue, generates SQL exclusively via the `nl-sql` skill, executes queries via Kyuubi/Spark, generates local CSV + chart files, runs multi-expert analysis, and publishes a complete report to a Feishu document.

## Phase 1: Requirement Clarification

The Main Agent engages in multi-turn dialogue to pin down the analysis request.

- Inputs: user's natural language question + conversation history
- Agent asks clarifying questions one at a time
- Maps user intent to nl-sql's three elements:
  - **Business line**: 浏览器主端 / 浏览器信息流 / 内容中心 / 搜索 / 小说
  - **Metric**: DAU, MAU, retention, duration, ARPU, event tracking, etc.
  - **Dimensions**: date, new/old user, launch method, model, version, province, etc.
  - **Time range**: specific dates or relative (last 7 days, last month, etc.)
  - **Data type**: core metrics vs. event tracking
- Outputs a confirmed analysis brief before proceeding

The agent must get explicit user confirmation before moving to Phase 2.

## Phase 2: SQL Generation (nl-sql Skill — Exclusive)

**CONSTRAINT: All SQL must be generated through the nl-sql skill. The agent must NOT write SQL directly under any circumstance.**

### nl-sql Execution Flow

The agent delegates to `nl-sql` skill which follows this strict process:

1. **Confirm business line** → maps to reference file directory
2. **Confirm data type** → core metrics or event tracking
3. **For core metrics (3A flow)**:
   - Collect metric name, time range, dimensions from Phase 1 output
   - Load reference files: `reference/{business_line}/metric-name-index.md`, `metric-dimension-index.md`, `core-metrics-tables.md`
   - Map to standard metric (e.g., "日活" → DAU → `BM001` → `is_app_dau_2024=1`)
   - Map dimensions to SQL fields (e.g., "新老用户" → `is_new_2024`)
   - Apply three-element combination: metric logic (fixed) + dimensions (flexible) + time range
   - Self-check: date format `YYYYMMDD`, field validity, table name `iceberg_zjyprc_hadoop.*`, partition filter
   - User confirms mapping before SQL generation
4. **For event tracking (3B flow)**:
   - Load event name index: `reference/{business_line}/event-name-index.md`
   - Generate query against `dwd_ot_event_di_31000000442` table
   - UV (`COUNT(DISTINCT distinct_id)`) + PV (`COUNT(*)`) as default metrics
5. **Save SQL file** via `python nl-sql/scripts/nl-sql.py --metric "{name}" --sql "{sql}"`

### Output

- SQL file saved to `~/Desktop/{YYYYMMDD}_{HHMMSS}_{metric}/`
- SQL content available for Phase 3 execution

## Phase 3: Data Query & Visualization

### Query Execution

- Use `kyuubi` CLI to execute the SQL generated by nl-sql
- `kyuubi sql query "{sql}"` → returns result rows
- For read-only SELECT queries, no confirmation needed
- Results parsed into structured data (pandas DataFrame)

### Output Files

- CSV: saved to `./output/{timestamp}_{topic}.csv`
- Charts: Matplotlib generates PNG files to `./output/{timestamp}_{topic}_{chart_type}.png`
- Chart types determined by data shape: bar, line, pie, heatmap, scatter

## Phase 4: Multi-Agent Analysis

After data is collected and visualized, the Orchestrator spawns three expert sub-agents. Each receives the same context (query results, chart descriptions, analysis brief) but has a distinct system prompt and role.

### Expert Roles

| Role | System Prompt Focus | Input | Output |
|------|-------------------|-------|--------|
| SQL Engineer | Data quality, query correctness, schema expertise | Raw SQL (from nl-sql) + result sample + table schemas | Data quality assessment, data lineage notes, supplementary query suggestions |
| Data Analyst | Statistical methods, trend detection, anomaly finding | Full query result + chart metadata | Statistical summary, trend analysis, anomaly flags, correlation insights |
| Business Advisor | Domain knowledge, actionable insights, risk awareness | Query results + Data Analyst's preliminary findings | Business interpretation, action recommendations, risk warnings |

### Execution Flow

1. SQL Engineer and Data Analyst run in parallel (both only need raw data)
2. Business Advisor runs after receiving Data Analyst output (needs analytical context)
3. Orchestrator merges all three outputs into a structured report

### Sub-Agent Implementation

Each sub-agent is a separate `client.messages.create()` call with:
- A role-specific system prompt
- The shared data context (CSV summary, chart list, analysis brief)
- No tool access (pure reasoning, no side effects)

```python
async def run_expert(role: str, system_prompt: str, context: str) -> str:
    response = await client.messages.create(
        model="claude-sonnet-4-6",
        max_tokens=4096,
        system=system_prompt,
        messages=[{"role": "user", "content": context}]
    )
    return response.content[0].text
```

The Orchestrator uses `asyncio.gather()` for parallel execution of SQL Engineer + Data Analyst, then runs Business Advisor sequentially.

## Phase 5: Delivery

### Feishu Document

The Orchestrator composes a final markdown report and writes it to Feishu:

```bash
feishu docx create "{title}" -c "{markdown_content}"
```

Document structure:
1. Executive Summary (from Business Advisor)
2. Data Overview (table, time range, record count)
3. Key Findings (from Data Analyst)
4. Data Quality Notes (from SQL Engineer)
5. Recommendations (from Business Advisor)
6. Appendix: SQL queries used, chart images (described), data source metadata

The document URL is printed to the terminal.

### Output Artifacts

| Artifact | Location |
|----------|----------|
| SQL files | `~/Desktop/{YYYYMMDD}_{HHMMSS}_{metric}/` (via nl-sql) |
| CSV data files | `./output/{timestamp}_{topic}.csv` |
| Chart images | `./output/{timestamp}_{topic}_{chart}.png` |
| Feishu document | URL printed to terminal |

## Error Handling

- nl-sql metric not found: show available metrics from reference files, ask user to clarify
- nl-sql validation failure: show self-check errors, re-run generation
- Kyuubi query failure: show error to user, suggest re-running nl-sql with adjusted parameters
- Empty result set: inform user, suggest broader filters or different metric
- Feishu document creation failure: save report as local markdown fallback
- Sub-agent timeout: proceed with available expert outputs, note missing analysis

## Conversation History

- Stored in-memory as a list of `{role, content}` message dicts
- Passed to each new `messages.create()` call for multi-turn continuity
- Cleared on explicit user reset or new analysis topic
- History is NOT persisted across sessions (stateless between runs)

## Hard Constraints

1. **SQL generation is exclusively via nl-sql skill.** No hand-written SQL, no other SQL generation path.
2. **nl-sql's three-element model must be followed**: business line (fixed table) + metric (fixed filter/aggregation) + dimensions (flexible GROUP BY).
3. **nl-sql self-check must pass** before SQL is executed: date format, field validity, table name prefix, partition filter, metric definition consistency.
4. **Kyuubi is execution only.** It receives SQL from nl-sql, never generates SQL.
